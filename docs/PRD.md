# 产品需求文档 (PRD)：四人排班管理系统

## 一、 项目概述

- **项目名称**：部门人员排班管理系统（Web端）
- **项目目标**：为特定4人团队提供按周管理的排班工具，支持“手动排班+算法自动补全”，并能自动计算工作量、校验排班规则，最终将数据存储在本地。
- **技术基础**：纯前端项目，无后端数据库，所有数据和状态通过浏览器 `localStorage` 存储。

## 二、 核心功能与交互流程

1. **周排班表切换**：顶部提供“周选择器”，可以随意切换查看历史或未来的排班表。
2. **周末强制提醒**：系统检测当前系统时间，若在**周六或周日**打开系统，且本地缓存中没有**下一周**的排班数据，弹出醒目提示框（“下周暂无排班，请尽快安排！”）。
3. **手动排班与修改**：用户可在任意格子上进行手动排班操作（如下拉选择任务）。
4. **一键智能排班**：点击“自动生成”按钮，算法将在避开用户手动已排任务的前提下，按规则填满排班表。如果不满意，可无限次点击重新生成。
5. **保存与持久化**：点击“保存”按钮，当前周的数据写入 `localStorage`。保存后仍可继续修改。

## 三、 界面布局设计 (UI)

页面分为左右/上下两大核心区域：

### 1. 排班主表区

为一张二维表格，结构如下：

- **表头第1行**：日期（如：2023-10-23 到 2023-10-29）
- **表头第2行**：星期（周一 到 周日）
- **第一列**：人员名单（朱克捷、高琪、李敏欣、杨秀芬，共4人）
- **第二列**：上下午拆分（针对每个人拆分为“上午”、“下午”两行）
- **数据单元格**：具体的任务名称，**每个单元格（半天）最多只能排1个任务**，或者为“空（休假）”。

### 2. 工作量统计看板区

实时展示当前表格中4个人的周工作量统计：

- 列表展示4人各自的累计总工作量（保留1位小数）。
- 计算逻辑：**差值 = 最高工作量 - 最低工作量**。
- **UI高亮警报**：如果 差值 > 0.5，该看板区域或差值数字标红，提示工作量分配不均（仅做提示，不阻断保存）。

## 五、 排班算法规则 (核心业务逻辑)

当用户点击“自动生成”时，算法需严格遵守以下规则（**用户手动排入的任务优先级最高，算法需绕开已占用的格子**）：

### 1. 必排与限额规则

- **个人必排**：每个人每周必须被安排 **1次“电话”** 和 **1次“筛查”**（上下午皆可）。
- **部门必排**：整个部门每周必须安排 **1次“运动处方”**（必须在下午）和 **1次“舌苔评估”**（必须在上午）。
- **固定必排**：**周四下午**必须有 **2个人** 排“群石墨修改”；**周六下午**必须有 **1个人** 排“群石墨修改”。
- **留白规则**：排完所有必排任务后，剩余的时间槽用“随访”、“基础班”填充，允许存在“无任务（休假）”状态。

### 2. 连休与疲劳规则 (需跨周计算)

_注：跨周计算即排本周一、周二的班时，需读取 `localStorage` 中上周六、周日的数据。_

- **连续两休规则**：每人每周必须有连续的2天完全没有任务（上下午皆为空）不能多也不能少，刚好休息两天。
  - 合法情况示例：周二周三休、周六周日休，或者 **上周日+本周一** 休。休假是要包括上午和下午一起休
- **夜班防疲劳规则**：如果某人前一天的下午排了“随访下午/夜”，则其**第二天的上午必须为空**（不可排任何任务）。此规则需跨周生效（上周日影响本周一）。
- **连续上午防疲劳规则**：
  - **禁忌组合**：“随访上午”、“舌苔评估（上午）”、“门诊（无论上午下午，只要占用上午时段）”。
  - **限制**：以上三种任务**不可以连续两天排在上午**。（例如：周一排了舌苔评估，周二上午就不能排随访上午，也不能排门诊）。此规则需跨周生效。
  - **特例**：用户手动安排的“门诊”可以强行破坏此规则。但自动算法在填充其他任务时，若发现某人连续3天上午被手动安排了门诊，算法绝不在其前一天或后一天的上午排“随访上午”或“舌苔评估”。
  - **允许的组合**：如果周一排了“门诊上午”，周二上午排“电话”或“筛查”是**被允许的**。

## 六、 数据结构设计建议 (供开发参考)

推荐 `localStorage` 的存储结构为 JSON 格式，以周一的日期字符串作为 Key：

```json
// Key: "schedule_2023-10-23"
{
  "weekStartDate": "2023-10-23",
  "data": {
    "朱克捷": {
      "Monday": { "AM": "随访上午", "PM": "基础班" },
      "Tuesday": { "AM": "", "PM": "运动处方" },
      "Wednesday": { "AM": "", "PM": "" }
      // ...以此类推
    }
    // ...其他三人数据
  }
}
```

## 七、 异常与极端情况处理

1. **死锁处理**：由于排班规则较多（尤其是连休和跨周规则），自动生成算法可能遇到找不到可行解的情况（死锁）。若算法经过多次尝试（如循环1000次）仍未找到完全符合规则的排班表，应 `Toast` 提示：“条件过于严苛或已被手动任务占满，无法完全自动生成，请手动微调或减少预排任务。”
2. **缺省数据兼容**：如果跨周读取时发现“上周没有排班数据”，则默认上周六、周日人员为全员空闲，不应用跨周疲劳规则。
